The AiDSR evaluates **software system proposals whose primary goal is execution readiness, not inspiration or buy-in**.

The AiDSR rejects roadmap optimism, feature counting, market sizing, and technical bravado.  
Instead, it scores systems on their ability to **simulate lived reality with sufficient fidelity** such that implementation would add **scale**, not understanding.

Evaluation occurs **assuming simulation artifacts are already produced**, using a constrained rubric that favors **illumination, bounded uncertainty, and structural coherence** over velocity or ambition.

---

## Scoring Overview

Systems are scored across **10 dimensions (0–10)** for a total of **0–100**.

**Decision thresholds**

- **< 60** — Reject (SITD likely; discovery deferred to build)
    
- **60–69** — Plausible but under-simulated
    
- **70–79** — Coherent system; execution still risky
    
- **80–89** — High-fidelity simulation; execution justified
    
- **90–100** — Inevitable system; build is clerical
    

---

## Dimension 1: Post-Launch Reality Clarity

**Question**  
Is daily life _after launch_ clearly imagined and articulated?

**Scoring guidance**

- 0–2: Abstract benefits only
    
- 3–5: Vague user stories
    
- 6–8: Concrete behaviors and rhythms
    
- 9–10: Lived reality feels already inhabited
    

If the system cannot be described as ongoing life, subtract points.

---

## Dimension 2: Pre-Implementation Illumination

**Question**  
How much understanding exists **before** any code is written?

**Scoring guidance**

- 0–2: Understanding expected to emerge during build
    
- 3–5: Partial clarity, major unknowns implicit
    
- 6–8: Decisions documented; gaps enumerated
    
- 9–10: No discovery pressure remains
    

Surprises during build indicate low scores retroactively.

---

## Dimension 3: Simulation Completeness

**Question**  
How many production-grade artifacts exist _as if the system were live_?

**Scoring guidance**

- 0–2: Concept only
    
- 3–5: Fragmented docs or mockups
    
- 6–8: End-to-end simulation artifacts
    
- 9–10: System could be taught, sold, and onboarded today
    

Absence of education artifacts caps score at 6.

---

## Dimension 4: Behavioral Contract Definition

**Question**  
Are system behaviors and boundaries explicit?

**Scoring guidance**

- 0–2: Behavior inferred from features
    
- 3–5: Happy paths only
    
- 6–8: Edge cases and failure modes named
    
- 9–10: Behavioral contracts are explicit and testable
    

Ambiguity defaults to the lowest plausible score.

---

## Dimension 5: Bounded Unknowns

**Question**  
Are uncertainties explicitly surfaced and constrained?

**Scoring guidance**

- 0–2: Unknowns ignored or deferred
    
- 3–5: Risks acknowledged informally
    
- 6–8: Unknowns listed with scope
    
- 9–10: Unknowns are finite and non-structural
    

Unexamined unknowns invalidate high scores elsewhere.

---

## Dimension 6: Explanation Survivability

**Question**  
Can the system be explained repeatedly without degradation?

**Scoring guidance**

- 0–2: Requires persuasion or insider context
    
- 3–5: Explanation varies by narrator
    
- 6–8: Stable explanation across audiences
    
- 9–10: Explanation teaches itself
    

If meaning decays with repetition, subtract points.

---

## Dimension 7: Delivery Discipline Requirement

**Question**  
Does the system demand restraint rather than heroics?

**Scoring guidance**

- 0–2: Success depends on exceptional execution
    
- 3–5: Requires constant judgment calls
    
- 6–8: Clear constraints guide execution
    
- 9–10: Discipline enforces itself
    

Systems that rely on talent instead of structure underperform.

---

## Dimension 8: SITD Resistance

**Question**  
How resistant is the system to Sprinting Into the Dark?

**Scoring guidance**

- 0–2: Meaning emerges only after shipping
    
- 3–5: Partial illumination
    
- 6–8: Clear illumination gates exist
    
- 9–10: SITD is structurally impossible
    

If velocity can outrun clarity, score ≤5.

---

## Dimension 9: Promotion Parity

**Question**  
Does external messaging merely mirror the simulation?

**Scoring guidance**

- 0–2: Narrative invented for launch
    
- 3–5: Messaging loosely aligned
    
- 6–8: Promotion restates known reality
    
- 9–10: Promotion feels redundant
    

If marketing must invent meaning, subtract points.

---

## Dimension 10: Execution Inevitability

**Question**  
Does building feel like the obvious next step?

**Scoring guidance**

- 0–2: Build framed as exploration
    
- 3–5: Execution justified emotionally
    
- 6–8: Build is operationally sensible
    
- 9–10: Not building feels negligent
    

Inevitability is the ceiling condition.

---

## Rapid Elimination Filter

Immediate rejection if the system:

- Uses build as a learning mechanism
    
- Defers explanation to “after MVP”
    
- Treats documentation as optional
    
- Confuses motion with progress
    
- Relies on user feedback to define purpose
    

---

## Closing Principle

Execution is not a reward for ambition.  
It is a responsibility earned through **simulation fidelity**.

A system scores highly only when it can be treated **as if it already exists**—and nothing important is left to discover once code begins.
